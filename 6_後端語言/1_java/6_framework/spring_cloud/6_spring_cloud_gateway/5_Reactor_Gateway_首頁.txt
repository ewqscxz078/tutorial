Spring Cloud Gateway (WebFlux 版) 裡，Gateway 本身不是傳統 MVC 應用，它沒有「首頁」或靜態頁面的概念。

所以當你：
	curl -i http://localhost:9001/
		去打 Gateway 的根路徑 / 時，因為 沒有任何 route predicate 匹配 /，
		Gateway 會直接回 404 Not Found —— 這是正常行為。

有幾個選項
	1.加一個首頁 route
		如果你想打 / 也能有東西回，可以在 application.yml 補一個 route，讓 / 直接轉發到某個後端服務：
			spring:
			  cloud:
				gateway:
				  routes:
					- id: root-home
					  uri: http://localhost:8080   # 你的後端首頁服務
					  predicates:
						- Path=/
	
	2.自訂 Fallback Handler
		寫個 @RestController 或 HandlerFunction<ServerResponse>，處理沒有匹配 route 的請求。
			@RestController
			public class HomeController {
				@GetMapping("/")
				public String home() {
					return "Spring Cloud Gateway is running";
				}
			}
		這樣 / 就有個訊息回應，而其他路徑還是交給 Gateway route。
	
	3.讓 / 自動跳轉
		如果你只是希望 / 自動跳到某個 gateway 路徑，可以寫一個 RedirectFilter 或簡單在 HomeController 裡：
			@GetMapping("/")
			public Mono<Void> redirect(ServerHttpResponse response) {
				response.setStatusCode(HttpStatus.FOUND);
				response.getHeaders().setLocation(URI.create("/your/api"));
				return response.setComplete();
			}
	👉 總結：
		Spring Cloud Gateway 預設沒有首頁，打根目錄 / 會 404 是預期的。
		如果你要有首頁/健康檢查/轉跳，得自己補上一個 route 或 controller。

適合情境
	適合 補一個 route 的情境
		* 你希望 / 直接代理到某個「真實後端」的首頁（例如 UI、Landing Page、Docs）。
		* Gateway 本身不該回應內容，只是「轉送」。
		* 好處：保持 Gateway 的職責單純（純粹做轉發）。
		* 壞處：如果後端掛掉，連 Gateway / 也會變 502。

	適合 補一個 controller 的情境
		* 你只是要讓 / 不要回 404，回一個健康訊息或簡單 JSON。
		* 你想讓 / 用來健康檢查（例如 k8s liveness probe）。
		* 好處：Gateway 啟動就能直接回應，不依賴後端。
		* 壞處：Gateway 本身開始多做「非轉發」的事，職責稍微擴張。

	✅ ChatGPT建議：
		如果 / 是 對外入口的首頁（UI、Docs），請用 route → 保持 Gateway「只轉送」的定位。
		如果 / 只是 健康檢查或佔位訊息，請用 controller → 簡單快速、不依賴後端。
