Q: 一般設定 spring.application.name 會跟 web 的 context-path 一致嗎?
A:	不一定，而且通常不建議把兩者綁死。

	在 Spring Boot / Spring Cloud Config 的語意裡，這兩個東西本質不同：
		* spring.application.name：應用識別名
			主要用在 Spring Cloud Config 取設定（{application}-{profile}.yml）、服務識別、Actuator、觀測/metrics tag、某些 auto-config 行為等。

		* server.servlet.context-path（或 reactive 的 spring.webflux.base-path）：HTTP 路徑前綴
			影響 URL 長相，例如 /order、/billing 這種。

	所以「會不會一致？」答案是：你可以讓它一致，但框架不會要求，也不會自動幫你保持一致。

	什麼情境會想讓它一致

		* 你們團隊把「服務名」就是「對外 URL 前綴」當成規範（例如都用 /order 對應 order-service）。

		* 你希望 config repo 的檔名、K8s service name、監控維度、以及 URL 路徑都看起來一致，降低認知負擔。

		但就算要一致，實務上也常見是「同一個根名字」一致，而不是字串完全一樣，例如：

			* spring.application.name=order-service

			* server.servlet.context-path=/order

	什麼情境不建議一致
		* 你有 多個 app 共享同一個 gateway context-path（例如 gateway 轉發 /order/** 到不同版本 order-v1, order-v2），這時 application.name 需要更精準。

		* 你會調整 URL 結構（版本化、重構路由）但不想連 Config 檔名與服務識別也跟著改（改名牽動 config 檔名、部署、監控維度）。

		* 同一個應用可能以不同 context-path 部署在不同環境（很常見），但 config 仍希望沿用同一份 application name 的設定基底。

	給你一個實務建議（最常用、也最不會踩雷）
		* spring.application.name：用「服務識別」命名（穩定、跟部署/監控一致）
			適合不變動

		* context-path：用「對外 API 路徑」命名（可演進、可版本化）
			可能不同情境使用不同 context-path

		例如：
			spring:
			  application:
				name: order-service

			server:
			  servlet:
				context-path: /order

		如果你後面要玩 Spring Cloud Config（2025.0.1）更舒服：config repo 檔名也會更清楚：
		order-service-dev.yml, order-service-prod.yml 之類。
